## 1. 消息队列基础概念

### 1.1 什么是消息队列

消息队列（Message Queue）是一种**异步通信**的中间件，让"发送方"和"接收方"不直接打交道，通过一个中间人（队列/代理）传递数据。

```
┌──────────┐   put(msg)    ┌────────┐   take()    ┌──────────┐
│ Producer │ ───────────▶  │ Queue  │ ──────────▶ │ Consumer │
└──────────┘               └────────┘             └──────────┘
   生产者                  队列 / Broker            消费者
```

### 1.2 核心思想：解决四个问题

| 思想 | 解决的问题 |
|------|-----------|
| **解耦（decouple）** | 双方无需知道对方的地址、协议、是否在线 |
| **异步（async）** | 生产者发完就走，不阻塞等结果 |
| **削峰（buffer）** | 突发流量先堆在队列里，消费者慢慢处理 |
| **可靠（reliable）** | 消息持久化，重启/宕机后仍能投递 |

### 1.3 四个核心抽象

```
1. Message      一段字节（你写啥都行），通常带 topic / key / headers
2. Producer     send(message)
3. Broker       保存消息，负责投递
4. Consumer     poll()  或  回调 onMessage(message)
```

### 1.4 三组关键设计抉择

#### ① 投递语义

| 语义 | 说明 | 风险 |
|------|------|------|
| **At-most-once** | 发完就忘，可能丢 | 可能丢失消息 |
| **At-least-once** | 没确认就不删，可能重复 | 消费端必须幂等 |
| **Exactly-once** | 靠幂等生产+幂等消费+事务组合 | 实现复杂 |

#### ② Push vs Pull

| 模式 | 特点 |
|------|------|
| **Push（主动推）** | 实时高，但 broker 不知道消费者扛不扛得住 |
| **Pull（主动拉）** | 消费者掌控节奏，可批量、容易背压。Kafka/RocketMQ 都走 pull |

#### ③ 队列模型 vs 发布订阅

| 模型 | 特点 |
|------|------|
| **Queue（点对点）** | 一条消息只被一个消费者消费 |
| **Topic（发布订阅）** | 一条消息被所有订阅者看到 |

### 1.5 最简单的代码实现

```java
import java.util.concurrent.*;

public class TinyMQ {

    private final BlockingQueue<String> queue = new LinkedBlockingQueue<>(1024);

    // 生产者：阻塞直到队列有空位
    public void produce(String msg) throws InterruptedException {
        queue.put(msg);
    }

    // 消费者：阻塞直到有消息
    public String consume() throws InterruptedException {
        return queue.take();
    }

    // === demo ===
    public static void main(String[] args) {
        TinyMQ mq = new TinyMQ();

        // 一个生产者
        new Thread(() -> {
            int n = 0;
            try {
                while (true) {
                    mq.produce("task-" + n++);
                    Thread.sleep(100);
                }
            } catch (InterruptedException e) { return; }
        }, "producer").start();

        // 三个消费者（并行处理）
        for (int i = 0; i < 3; i++) {
            new Thread(() -> {
                try {
                    while (true) {
                        String msg = mq.consume();
                        System.out.println(Thread.currentThread().getName() + " got " + msg);
                    }
                } catch (InterruptedException e) { return; }
            }, "consumer-" + i).start();
        }
    }
}
```

> **核心记忆**：消息队列的所有"高大上"特性，本质都是给这条最基本的 `put/take` 加约束。

### 1.6 从最小到生产级的演进

```
最小 (30行)          生产级 (Kafka / RocketMQ)
─────────────────    ──────────────────────────────
内存队列        →     顺序写磁盘 + mmap
无副本          →     leader-follower 多副本 + ISR
单队列串行      →     partition + consumer group 并行
拿到就算成功    →     offset + ACK + DLQ
没超限就好     →     backpressure / quota
无序           →     partition 内有序 (key hash)
单实例         →     controller 选举 + controller quorum
```

## 2. MQ 与数据库的关系

### 2.1 本质：状态 vs 事件

> **数据库 = 把状态持久化下来；MQ = 把"状态变化的事件"持久化下来。**

| 层 | 谁负责 | 长什么样 |
|----|--------|---------|
| **当前状态（state）** | 数据库 | users 表 / orders 表，行级 CRUD |
| **事件流（event log）** | 消息队列 | `OrderCreated{user=1, sku=2, ts=...}` 的时间序列 |

### 2.2 存储结构对比

MQ 的存储结构和数据库 WAL 本质一样——都是 append-only log：

```
Kafka segment 文件：
├── segment-00000.log    ← 一行行 append 字节
├── segment-00001.log    ← 一个 64MB 文件就切到下一个
└── segment-00002.log

MySQL redo log / PostgreSQL WAL：
└── 00000000000000001.wal  ← 同样只追加的字节流
```

### 2.3 Outbox 模式：解决双写一致性

把消息**先和业务数据写在同一个数据库事务里**，再由后台程序"搬"到 MQ：

```
                    ┌─────────────┐
                    │   客户端     │
                    └──────┬──────┘
                           │ 1. POST /order
                           ▼
┌──────────────────────────────────────────┐
│  订单服务                                   │
│  (1) BEGIN TX                              │
│  (2) INSERT INTO orders ...               │
│  (3) 将 OrderCreated 写入 outbox 表         │   ←★ 关键
│  (4) COMMIT                               │
└─────────────────────┬─────────────────────┘
                      │
                      ▼
┌──────────────────────────────────────────┐
│  outbox-poller (定时任务 / Debezium)        │
│  把 outbox 表的新行 → 投递到 Kafka          │
└─────────────────────┬─────────────────────┘
                      │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
   库存服务       营销服务       数据仓库
```

### 2.4 三种典型协作模式

| 模式 | 谁先写 | 解决问题 | 代表场景 |
|------|------|---------|---------|
| **DB 为主，MQ 为辅** | 先写 DB，再发 MQ | 普通业务通知 | 下单成功发短信 |
| **MQ 为主，DB 缓存** | 先发 MQ，消费者异步落 DB | 写吞吐极大 | 日志采集、点击流、IoT |
| **outbox（双写一致）** | DB 里同事务写消息 | 既要不丢，又要解耦 | 金融交易、订单状态 |

### 2.5 架构全景图

```
               ┌─────────────────┐
               │   客户端 / API    │
               └────────┬────────┘
                        │
              ┌─────────▼─────────┐
              │    业务服务         │ ──────▶ 读 cache 层（Redis）
              │   (写 DB + outbox)  │
              └────┬───────────┬───┘
                   │           │
                   ▼           ▼
        ┌──────────────┐  ┌──────────────────┐
        │   关系数据库   │  │     outbox 表     │ ──► Debezium/Poller ──► Kafka
        │  (当前状态)   │  │ (同事务的消息)      │                  │
        └──────────────┘  └──────────────────┘                  │
                   ▲                                            │
                   │           重放事件流重建                     │
                   └────────────────────────────────────────────┘
                                                                 │
                                            ┌────────────────────┼────────────────────┐
                                            ▼                    ▼                    ▼
                                       搜索/分析服务         通知/营销服务          数据仓库
```

## 3. 调度驱动 vs 事件驱动

### 3.1 一句话对比

> **调度驱动**：我决定现在该干活了，去把活儿做掉。
> **事件驱动**：现实里发生了某件事，系统被动地因此去响应。

### 3.2 流程图对比

**调度驱动**：
```
              调度器（"每一分钟跑一次"）
                     │
                     ▼
              ┌──────────────┐
              │  业务方法执行  │   ← 谁来调用决定何时触发
              └──────┬───────┘
                     │
                同步/阻塞等结果
                     │
                     ▼
              ┌──────────────┐
              │   继续后续逻辑  │
              └──────────────┘
```

**事件驱动**：
```
   事件源            事件总线           监听者们
 ┌────────┐        ┌─────────┐     ┌─────────┐
 │ 产生事件 │──▶──▶│   Event  │────▶│ Listener A│
 └────────┘  publish  Bus     │────▶│ Listener B│
                            │     └─────────┘
                            └────▶│ Listener C│
                                  └─────────┘
                         谁关心谁就来订阅
```

### 3.3 核心差异表

| 维度 | **调度驱动** | **事件驱动** |
|------|------------|------------|
| 触发者 | **调用方**主动触发 | **事件发生**自动触发 |
| 时间 | 何时执行提前定好（定时/轮询） | 何时发生不可预测 |
| 依赖 | 调用方要知道被调者 | 双方互不知道 |
| 错位处理 | 失败要重试，重试要 idempotent | 自动可重试，可丢到 DLQ |
| 扩展性 | 加下游要改调用方代码 | 加消费者只多订阅一次 |
| 失败影响 | 一处失败可能阻塞上游 | 失败只影响局部 |
| 调试难度 | 容易（同步栈追踪清楚） | 难（异步链路散开） |

### 3.4 各自适合的场景

#### 调度驱动更适合的：

| 场景 | 为什么 |
|------|------|
| **离线批处理**（清算、对账） | 必须定时跑，不用关心谁去触发 |
| **明确的依赖链**（订单→支付→发货） | 必须按顺序、有因果关系 |
| **周期性管理任务**（清理日志、刷新缓存） | 时间是主语，事件是次要的 |
| **需要严格顺序 + 不希望延迟** | 不引入异步，顺序最清楚 |
| **强事务** | 同步事务边界一目了然 |

#### 事件驱动更适合的：

| 场景 | 为什么 |
|------|------|
| **不可预测的输入**（用户点击、IoT 异常、订单支付回调） | 谁也不知道用户下一秒会干嘛 |
| **一个事件要让多个系统反应**（下单后通知、积分、推荐） | 一次事件 → 多个消费者 |
| **业务链路长、可能被中途失败打断** | 失败可重试，不用调用方扛 |
| **性能 / 解耦是首要诉求** | 异步 + 缓冲天然就合适 |
| **需要回放历史**（审计 / 重算） | 事件流天然就是"录像带" |

### 3.5 案例对比 —— "下单后给用户加积分"

**调度驱动写法**：
```java
@Transactional
public void placeOrder(...) {
    orderRepo.save(order);
    pointsClient.addPoints(userId, 100);   // ← RPC
    couponClient.grant(userId, "WELCOME"); // ← RPC
}
```
问题：**积分挂了，订单回滚。**

**事件驱动写法**：
```java
@Transactional
public void placeOrder(...) {
    orderRepo.save(order);
    eventBus.publish(new OrderPlaced(userId, 100));   // 一句话
}
```
积分服务、营销服务各自订阅 `OrderPlaced`。**它们全挂了用户也下单成功。**

### 3.6 真实世界的混合体

```
用户点击 → 事件驱动进系统
            │
            ▼
        订单服务
            │ ─── 同步 RPC：调用支付（要等结果）
            │
            ▼
        落库 + 发 OrderPaid 事件
            │
            ▼
        调度任务：每晚 02:00 跑对账
```

## 4. MQ 高吞吐原理

### 4.1 总体思路

> **放弃"随机读、事务、复杂查询"这三件事，把硬件压榨到极限。**

```
软件层 ─┬── 批量化 / 异步化 / 顺序化
        ├── 零拷贝 / 内存映射
        └── 分区并行
       ─┐
OS 层   ├── page cache / 文件系统 / sendfile
       ─┘
硬件层 ─── 顺序写盘 / 大页 / CPU cache 友好
```

### 4.2 批次化（Batching）

把 1 万次 syscall 变成 10 次：

```java
// 反例：每条消息都触发一次系统调用
for (msg : msgs) {
    File.write(msg);   // 1 万次系统调用 = 性能噩梦
}

// MQ 的做法：攒一批再 flush
while (running) {
    batch = poll();          // 攒到 16KB 或 50ms
    file.append(batch);      // 一次 write，一次刷盘
}
```

### 4.3 顺序写磁盘 —— 磁盘的"甜区"

| 介质 | 随机写 | **顺序写** |
|------|-------|-----------|
| HDD (7200 RPM) | ~100 IOPS | **~100 MB/s ≈ ~50000 IOPS** |
| SSD | ~10000 IOPS | **~500 MB/s ≈ ~100000 IOPS** |
| 内存 | 数十万 IOPS | 数十万 IOPS |

MQ 强制消息按 offset 顺序 append，**永远不修改、不随机读**——磁盘顺序写接近内存速度。

### 4.4 零拷贝（Zero-copy）

**传统网络发送（4 次拷贝）**：
```
磁盘 ──▶ page cache ──▶ 应用 buffer ──▶ socket buffer ──▶ 网卡
        (DMA)           (CPU 拷贝)        (CPU 拷贝)
```

**MMAP + sendfile 的零拷贝（0 次 CPU 拷贝）**：
```
磁盘 ──▶ page cache ═════════════════════▶ 网卡
        (DMA)            (sendfile 内核态)
```

### 4.5 Page Cache 利用 —— 让 OS 当免费缓存

> 现代 Linux 默认会用空闲 RAM 当 page cache。Kafka 的 segment 文件直接 mmap 到 page cache，消费者读"几乎都在内存里"。

这就是为什么 Kafka 官方说：**"You don't need to use any cache layer in front of Kafka."** —— OS 已经干了。

### 4.6 分区（Partition）+ 顺序读

```
Topic "orders"
    ├── P0 ─→ 生产者 A、B
    ├── P1 ─→ 生产者 C、D
    └── P2 ─→ 生产者 E、F
```

- 写入压力被平摊到 N 个文件、N 个线程
- 消费者也按 partition 数量并行
- 一个 partition = 一次 append-only = 单线程无锁顺序写 → 性能最高

### 4.7 单线程写 + 无锁

```java
class PartitionAppender {
    private final AtomicLong offset = new AtomicLong(0);
    public void append(byte[] msg) {
        long pos = offset.getAndIncrement();   // 原子分配序号
        file.write(pos, msg);                  // 单线程，无锁
    }
}
```

**写动作是"原子取序号 + 顺序写"。零锁、零竞争。**

### 4.8 拉（Pull）模式 —— 消费者掌控节奏

MQ 让消费者自己拉：

```java
// 消费者侧：攒一批再 ACK
while (running) {
    msgs = broker.fetch(minBytes=64KB, maxWait=500ms);
    for (m : msgs) { handle(m); }
    ack(msgs.last());
}
```

消费者可以：
- **攒够一批再处理**（网络包效率高）
- **处理不过来就让 fetch 等着**（天然背压）
- **断网回来一次性 batch 拿回**（不丢消息）

### 4.9 各种小优化合集

| 技巧 | 干什么用 |
|------|---------|
| **Compression（LZ4/ZSTD）** | 网络和盘 IO 同时降一半 |
| **Compact topic** | 老消息覆盖替代清理 |
| **稀疏索引**（offset→文件位置） | 不用扫整段 |
| **Page cache 预热** | 重启不丢缓存 |
| **大页（Hugepage）** | mmap 时 TLB miss 减少 |
| **避免对象分配** | 字节 buffer 池化复用，GC 压力小 |

## 6. 消息队列的不同形态

> **任务队列也是消息队列的一种表现形式吗？** ——是的。广义上一切"生产 → Broker → 消费"的中间层都叫消息队列；不同场景下，队列的语义、数据结构、API 设计差异很大，于是业界分化出多种"特化形态"。

### 6.1 按架构形态划分

#### ① 任务队列（Task Queue）

- **典型代表**：Celery（Python）、Sidekiq（Ruby）、Bull（Node）、Amazon SQS、Beanstalkd
- **典型场景**：发邮件、转码、定时清理、定时通知
- **典型消息格式**：`{ task: "send_email", args: [...], retry: 3, eta: ... }`
- **核心特征**：
  - 每条消息 = **一个待执行的任务**（"做什么"远重于"发生了什么"）
  - 通常配合**结果回执**（result backend：Redis/DB）记录 `task_id → 状态`
  - 多 worker **竞争抢任务**（点对点），保证任务只被一个 worker 跑
  - 任务生命周期：`queued → running → success / failed / retried`
  - 有成熟的失败重试、确认超时、状态查询等 API

> **vs 普通消息队列**：Kafka/RocketMQ 关心的是"事件该怎么流转"，任务队列关心的是"活儿该谁干、干没干完"。

#### ② 日志型事件流（Log-based Event Stream）

- **典型代表**：Kafka、Apache Pulsar、AWS Kinesis、Redpanda
- **典型场景**：用户行为采集、应用日志、订单事件流、ClickStream、IoT
- **核心特征**：
  - **磁盘即真相**（commit log / append-only segment），消息可重读
  - **按 offset 消费**而不是"取出来就删"，可以在不同时刻重复读同一段
  - **分区 + 水平扩展**：一个 topic 内有 N 个 partition，consumer group 内最多 N 个并行
  - **长保留期**：默认保留几天甚至永久（segment 滚出而不是"消费就删"）
  - **无内置"任务"概念**：消息只是字节流，**消费者自己决定语义**

> **这就是 P0 已经学过的"日志型事件流"那一类**。它和任务队列最本质的区别是：**消息会留在磁盘里被反复读**，不是为了执行一次性任务。

#### ③ 发布订阅 / 消息总线（Pub/Sub Message Bus）

- **典型代表**：Redis Pub/Sub、Google Cloud Pub/Sub、AWS SNS、NATS、RabbitMQ Topic Exchange
- **典型场景**：通知广播、配置变更推送、聊天消息、IoT 设备指令
- **核心特征**：
  - 一条消息 → **所有订阅者各收到一份**（扇出 fanout）
  - 通常订阅者要**在线**才收得到；离线状态容易丢（这一点和上两者显著不同）
  - 投递策略常常是"**尽快推送 + 不持久化**"（push 模式）
  - 不强调"任务执行"，强调"状态改变了一件事发生了"

#### ④ 内存级高性能队列（In-Memory High-Throughput Queue）

- **典型代表**：LMAX Disruptor、Aeron、Java `BlockingQueue` / JCTools
- **典型场景**：线程间通信、单进程内的超高吞吐链路（如交易所、游戏服务器）
- **核心特征**：
  - **不是分布式中间件**，而是**单进程内的"超快队列"**
  - 通过 ring buffer + 单线程写 + 无锁设计达到千万级 TPS
  - 这就是 P2 要去读 Disruptor 源码的那一类

#### ⑤ 事件存储（Event Store）

- **典型代表**：EventStoreDB、Axon Server（Kurrent），乃至把 Kafka 当事件存储
- **典型场景**：Event Sourcing 架构、审计回放
- **核心特征**：
  - **事件即真相**：不存当前状态，只存发生过的事件
  - 支持 stream（聚合根）维度的追加、订阅、投影
  - 把消息队列 + 数据库的能力合二为一

### 6.2 特征型队列（在基础 MQ 上叠加的"特殊形态"）

下面这些不一定是独立产品，但每一种都有清晰的使用场景，几乎所有 MQ 都会提供对应实现：

| 类型 | 核心特点 | 代表实现 |
|------|---------|---------|
| **延迟队列 / 定时队列** | 消息到了指定时间才可被消费 | RabbitMQ delayed-exchange、RocketMQ 18 个延迟级别、Redis `ZSET` + 扫描 |
| **死信队列（DLQ）** | 消费失败 N 次后丢到这里，避免阻塞主队列 | RabbitMQ DLX、RocketMQ DLQ、Kafka 死信 topic |
| **优先级队列** | 高优先级消息插队先消费 | RabbitMQ priority queue、ActiveMQ |
| **顺序消息队列** | 同一 key 的消息必须严格按发送顺序被消费 | RocketMQ 顺序消息、Kafka 单 partition |
| **事务消息队列** | "本地事务 + 消息发送"原子化（半消息 + 回查） | RocketMQ 事务消息、Kafka transaction API |
| **请求-应答队列（RPC-MQ）** | 用消息队列做 RPC：带 correlationId、临时响应队列 | RabbitMQ `reply-to`、gRPC + MQ |
| **广播消息** | 每条消息推给**同组内所有消费者**各一份 | RocketMQ 广播模式、Kafka broadcast |
| **CDC 队列** | 把数据库 binlog 转成事件流 | Debezium、Canal、Maxwell → Kafka |
| **流处理 / 状态化队列** | 消费者持有"状态"，在消息流上做聚合 | Kafka Streams、Flink、Materialize |

### 6.3 横向对比：为什么会有这么多"特殊"形态？

| 维度 | 任务队列 | 日志型事件流 | Pub/Sub 总线 | 内存高性能队列 |
|------|---------|------------|-------------|---------------|
| 一条消息被消费几次 | 1 次（worker 抢） | 多次可重放 | 一次（扇出但通常不持久） | 1 次 |
| 消费完删吗 | 是（典型实现） | **否**，保留 N 天 | 通常不持久化 | 是 |
| 关注重心 | "活儿干没干完" | "事件如何被多个系统看到" | "通知是否送达" | "传输极致快" |
| 顺序保障 | 通常单队列有序 | 同一 partition 有序 | 不承诺顺序 | 取决于实现 |
| 持久化 | 视实现 | **默认强持久化** | 多数不支持重读 | 通常无持久化 |
| 典型吞吐 | 中 | 极高（百万/秒级） | 中等（推送） | 极限（百万级/线程内） |
| 失败处理 | 自动重试 + DLQ | offset 自行提交，可重读 | 容易丢 | 通常无 |

### 6.4 小结

> 这些"特殊队列"是对通用抽象 **"解耦 + 异步 + 削峰 + 可靠"** 的极致强化：
> - 想干活儿 → **任务队列**（关心执行结果）
> - 想记流水 → **日志型事件流**（关心可回放）
> - 想广播通知 → **Pub/Sub 总线**（关心覆盖面）
> - 想超快 → **内存队列**（关心延迟和吞吐）
> - 想定时 → **延迟队列**；想高可靠 → **事务消息**；想隔离毒消息 → **DLQ**……

## 7. 为什么 Redis 也能做消息队列（广义 MQ 本质）

### 7.1 Redis 里能直接当 MQ 用的三套机制

| 机制 | Redis API | 本质是什么 | 等价于 |
|------|----------|----------|--------|
| **List 队列** | `LPUSH` + `BRPOP`（或 `RPUSH` + `BLPOP`） | 双端链表 + 阻塞读 | 一个分布式版的 `LinkedBlockingQueue` |
| **Pub/Sub** | `PUBLISH` / `SUBSCRIBE` | 频道订阅广播 | 实时消息总线（不带持久化） |
| **Stream**（5.0+） | `XADD` / `XREAD` / `XACK` + 消费组 | append-only log + 消费者组 + 确认机制 | **轻量级 Kafka** |

**为什么这些结构"刚好"就是队列？** 以最常见的 **List 队列**为例：

```
Producer ──LPUSH key msg──▶ [m3, m2, m1] ──BRPOP key──▶ Consumer
                          (队头是 m3，从 m1 这边出，FIFO)
```

- `LPUSH` 把消息塞到队头
- `BRPOP` 从队尾阻塞拿出（FIFO）
- 拿出来之前消息一直留在 List 里 → 天然 buffer
- 整个操作是 Redis **单线程串行执行** → 不需要锁也不会乱

### 7.2 Pub/Sub 和 Stream 的差异

```
Pub/Sub：
   发布 ──PUBLISH──▶ 频道 ──┬── 订阅者 A（在线才收得到）
                           └── 订阅者 B（离线 = 丢消息）

Stream：
   XADD ──▶ [m1, m2, m3...] ──▶ 消费组
                              ├─ 消费者 1：读到 m1~m2，XACK 之后才算完成
                              └─ 消费者 2：接着读 m3
                              └─ 消息持久化在内存 + RDB/AOF，重启不丢
```

> 注意：**Redis Pub/Sub 没有持久化，不算"可靠队列"**；要做可靠 MQ，要么用 List + 手动 ACK，要么用 Stream。

### 7.3 什么样"特性"的组件能当消息队列？

> **任何一个能"先存进去、稍后再拿出来交给别人"的数据结构，都可以是消息队列。**

#### 7.3.1 消息队列的本质三件套

```
        1. 一个缓冲区（buffer）
        2. 生产者可以往里塞消息（PUT）
        3. 消费者可以阻塞地取消息（TAKE / GET）
```

只要这三件事成立，就**满足最小定义**。

#### 7.3.2 把"广义 MQ"按核心特性分成 6 个维度

| 维度 | 极简档 → 高档 |
|------|--------------|
| ① **存储介质** | 进程内存 → 共享内存 → 磁盘文件 → 远程服务 |
| ② **取消息方式** | 进程内直接传引用 → 锁/条件变量 → socket 长连 → HTTP/gRPC |
| ③ **并发安全** | 单线程自然安全 → 加锁 → 无锁 CAS → 串行执行（Redis 单线程） |
| ④ **可靠性** | 不持久化 → 进程退出丢 → 写 WAL → 多副本复制 |
| ⑤ **消费模型** | 1 对 1（队列） / 1 对多（广播） / 多消费者负载均衡 |
| ⑥ **顺序保证** | 完全无序 → 单 key/单 partition 有序 → 全局强一致 |

#### 7.3.3 常见的"当 MQ 用"的组件对照

| 组件 | 用作 MQ 的方式 | 核心机制 | 优点 / 局限 |
|------|-------------|---------|-----------|
| **`LinkedBlockingQueue`**（Java） | 进程内 BlockingQueue | 锁 + 条件变量 | **最快最简单**，但只能进程内用 |
| **Redis List + BRPOP** | 分布式 BlockingQueue | LPUSH/BRPOP + 单线程串行 | 简单飞快，但没原生 ACK、堆积会撑爆内存 |
| **Redis Stream**（5.0+） | 轻量级 Kafka | append-only + 消费组 + XACK | 比 List 多了 ACK、消费者组、PEL；吞吐比 Kafka 低 |
| **PostgreSQL `SELECT ... FOR UPDATE SKIP LOCKED`** | 数据库当任务队列 | 行锁 + 跳过已锁行 | 强一致 + 事务；高并发下 DB 压力大 |
| **Kafka** | 专门的分布式日志型 MQ | 分段 append-only log + offset | 极致吞吐、可重放、可堆积 |
| **Beanstalkd** | 专门的任务队列 | tube + reserve + bury | 模型简单，但能力有限 |
| **RocketMQ / RabbitMQ / Pulsar** | 专门的 MQ 中间件 | 各家有差异但都满足①②③④⑤ | 功能齐全，最成熟 |

### 7.4 用一个统一视角看"为什么它们都能当 MQ"

把它们抽象成**同一张图**，本质都一样：

```
┌──────────────┐    ① 塞进去(PUT)     ┌──────────────────────────┐
│  Producer    │ ──────────────────▶ │    某块"可被取走"的存储      │ 
└──────────────┘                     │                            │
                                     │ · 进程内的堆/队列             │
                                     │ · Redis 的 List/Stream      │
                                     │ · Postgres 的一张表          │
                                     │ · Kafka 的 segment 文件      │
                                     │ · OS 的 pipe / socket       │
                                     └─────────────┬──────────────┘
                                                   │  ② 取出来(GET/TAKE)
                                                   ▼
                                             ┌──────────────┐
                                             │   Consumer    │
                                             └──────────────┘
```

**关键不在于"这个东西原本叫什么名字"，而在于：**
1. 能不能在 A 处塞进去 → 在 B 处取出来
2. 顺序是不是可控（FIFO 或可配置）
3. 塞进去和取出来是不是原子的（不会并发错乱）
4. 失败后能不能找回（持久化）

### 7.5 一个生动的对照实验

| "非典型组件" | 当 MQ 用 | 关键限制 |
|------------|---------|---------|
| **Socket** | `nc -l 1234` 一行命令 = 队列 | 没持久化、没多消费者 |
| **文件** | `producer 写文件` + `consumer tail -f` | 没 ACK、难重平衡 |
| **MySQL 表** | producer `INSERT`，consumer `SELECT ... SKIP LOCKED` | 强一致但慢 |
| **Redis Pub/Sub** | 在线广播 | 不持久化 |
| **Redis List** | 分布式 BlockingQueue | 没原生 ACK、需配合 `BRPOPLPUSH` 做可靠性 |
| **Redis Stream** | 准 Kafka | 吞吐不如 Kafka、分区能力弱 |

### 7.6 小结

> **任何"能存数据 + 阻塞地取数据"的组件，都能被当成消息队列。** Redis 之所以最容易做出来，是因为它**单线程串行执行 + 内存 + 现成的 List/Stream 数据结构**，天然就覆盖了"原子、阻塞、FIFO、可持久化"这四件事。
>
> 真正的差别不在"是不是 MQ"，而在**持久化、ACK、分区、多副本、堆积、回放** 这些加分项。加分项越多的，就是越专业的 MQ（Kafka/RocketMQ）；加分项少但简单的（Redis），就用在对可靠性要求没那么高的场景。

## 8. 幂等性与死信队列（DLQ）

### 8.1 幂等：MQ 可靠性的"必要条件"

> **幂等不是 MQ 本身的必要条件，而是"用 MQ 做出可靠业务"的必要条件。**

#### 投递语义和消费者要求

```
MQ 的"投递语义"           对消费者的要求
─────────────         ──────────────
At-most-once          不需要幂等（可能丢）
At-least-once         ★必须幂等（可能重复）
Exactly-once          ★必须幂等（即使承诺不重复，实现也靠幂等构造）
```

#### 为什么"At-least-once"这么常见？

几乎所有主流 MQ（RabbitMQ、Kafka、RocketMQ、Redis Stream）的默认行为都是 **"未确认就不删"**，目的是**优先不丢消息**。代价就是可能重复。

```
Producer ──▶ msg ──▶ Broker ──▶ Consumer
                                │
                          handle(msg)  ✓
                                │
                          ack(msg)  ✗ 网断了
                                │
                          Broker 不知道消费没成功
                          重发 → Consumer 又处理一次
```

**结果**：业务上要兜住"我可能执行 2 次这件事"。

#### 幂等的本质

> **同一个操作执行 N 次，最终结果和执行 1 次一样。**

```
幂等：  扣 1 元 → 扣 1 元 → 扣 1 元 = 只扣 1 次
不幂等：扣 1 元 → 扣 1 元 → 扣 1 元 = 扣了 3 次（完蛋）
```

#### 业界最常用的 4 种幂等套路

| 套路 | 怎么做 | 何时用 |
|------|------|------|
| **业务键去重** | 把"订单号 / 业务流水号"作为唯一索引，重复消息插入时冲突直接跳过 | 写数据库的场景 |
| **Redis SETNX/NX** | "我消费过的消息 ID" 存 Redis 里，重消费时先查 SET | 高频但需要快速判断 |
| **状态机判断** | 消息里带状态变迁 `paid → shipped`，当前状态对得上才执行 | 业务流程有清晰前驱 |
| **乐观锁/版本号** | `UPDATE ... WHERE version = old_version`，影响行数=0 说明已被处理过 | 任何"先读后写"的场景 |

> **MQ 不保证"只发一次"，业务自己保证"处理一次"。这就是"用 MQ 必须幂等"是金科玉律。**

### 8.2 死信队列 DLQ（Dead Letter Queue）

#### 8.2.1 死信 = "被抛弃但还想看看是不是真没救了"的消息

**一般定义**：一条消息在主队列里**达到失败上限 / 过期 / 被拒收**后，**不直接丢弃**，而是**转送到一个专门保留"坏消息"的队列**——这就是死信队列。

**触发"进入死信"的常见原因**：

| 触发条件 | 含义 |
|---------|------|
| `rejected` | 消费者用 `basicNack(requeue=false)` 显式拒绝（最常见） |
| `expired` | 消息 TTL 过期（队列级 TTL 或消息级 TTL） |
| `maxlen` / `quota` | 队列达到长度上限 |
| 消费者重试超限 | 业务侧自己约定"重试 3 次后转 DLQ" |

#### 8.2.2 一图看懂生命周期

```
                           reject / 失败超限
                                ▼
Producer ─▶ Main Queue ─▶ Consumer ─✗──▶ DLQ
   │            │            │
   │            │            └─ 失败 1 次：重试
   │            └─ 消息 TTL 到期
   ▼
（剩下没问题的消息正常消费）
                            ┌──────────┐
                            │ 运维/工具 │
                            └────┬─────┘
                                 ▼
                       人工查看 + 修复 + 重投
```

#### 8.2.3 死信队列通常怎么处理？

> **记住一句话：死信队列不是一个"丢弃场"，而是一个"隔离恢复场"。**

常见的 5 步处理流程：

| 步骤 | 干什么 |
|------|------|
| **① 落盘可观测** | DLQ 的消息要被持久化（不然白忙），最好**装监控告警**——DLQ 一增长就告警 |
| **② 不进主消费者** | DLQ **不能让主业务消费者再去拉**，否则死循环 / 业务混乱 |
| **③ 定期人工/工具排查** | 看消息内容、错误堆栈，定位根因（数据问题？逻辑 bug？下游挂了？） |
| **④ 修复 + 重投** | 通常用"重投工具"（RocketMQ 重投、Kafka 拷一份 offset 回去重做）回到主队列 |
| **⑤ 兜底数据校对** | 如果已造成业务不一致，还需要一个**对账补偿任务**（定时校正数据） |

#### 8.2.4 各家 MQ 的 DLQ 实现

| MQ | DLQ 实现 | 典型做法 |
|------|----------|----------|
| **RabbitMQ** | DLX（Dead Letter Exchange） | 配 `x-dead-letter-exchange`，消息自动路由到 DLX 绑定的队列 |
| **RocketMQ** | 专门的 DLQ 队列 | 重试次数超 `maxReconsumeTimes`（默认 5）后自动转 DLQ |
| **Kafka** | 没有内置 DLQ 概念 | 自己写：消费失败 → produce 到专门的 dead-letter topic |
| **Redis Stream** | 无内置 | 消费者丢弃 / 自己用另一个 Stream 记录 |
| **AWS SQS** | 内置 DLQ | 配 `RedriveAllowPolicy` 自动转死信 |
| **RabbitMQ + Spring** | `@RetryableTopic` | 注解一行实现"重试 + DLQ" |

### 8.3 小结

> **DLQ = 失败消息的"安全仓"。不是为了丢掉它，是为了让你"看得到、查得清、修得回"。**
>
> 配合**幂等消费**，构成"MQ 可靠消费"的标准拼图：
> - **幂等**：保证重复无害
> - **DLQ**：保证失败不丢、不阻塞主流程

## 9. 核心原则总结

> **时间能延后的，全部丢 MQ；必须同步等的，才留 RPC。**
> **状态写一次到 DB，事件写一份到 MQ；DB 是快照，MQ 是录像。**
