## 1. 我们真正抓住了什么

可以说，这次排查已经抓住了并发编程最核心的入口，但还不能把它简化成一句“多步骤操作要加锁”。更完整的认识是：

> 并发程序不是一些线程安全变量的集合，而是一套由多个线程共同执行的状态转换协议。

学习 CAS、volatile、锁、线程池和 Java 内存模型当然重要，但这些内容主要回答“底层机制提供什么保证”。在使用它们之前，我们必须先回答更上层的问题：

- 系统有哪些共享状态？
- 每个共享状态代表什么业务含义？
- 哪些状态必须一起变化？
- 什么条件在任何时刻都不能被破坏？
- 一个操作在哪个时刻才算真正完成？
- 某个线程停在任意一步时，其他线程还能不能继续推进？

如果这些问题没有答案，那么即使代码里到处都是 `AtomicLong`、CAS 和 volatile，也只能说明某些读写具有原子性或可见性，不能说明整个算法正确。

因此，并发设计的思考顺序应当是：

```text
业务语义
   ↓
共享状态与所有权
   ↓
必须始终成立的不变量
   ↓
完整状态转换和线性化点
   ↓
最坏线程交错
   ↓
安全性与可推进性证明
   ↓
锁、CAS、volatile、Atomic 等实现工具
```

并发工具是施工工具，不变量和并发协议才是设计图。

## 2. 案例回顾：每一步都“线程安全”，整体为什么仍然错误

多生产者写入 RingBuffer 的核心流程可以简化为三个步骤：

```java
long sequence = sequencer.next(); // 1. 申请序号
writeEntry(sequence, value);       // 2. 填写槽位
sequencer.publish(sequence);       // 3. 发布完成
```

单独观察每一步，很容易得到一种虚假的安全感：

- `next()` 使用 CAS，申请序号不会重复
- Entry 字段使用 volatile，消费者能够看到写入
- `publish()` 使用 volatile set，写入本身不会撕裂
- availableBuffer 使用 AtomicLongArray，不同生产者设置 bit 不会互相覆盖

然而，多线程执行的不是孤立的一行代码，而是不同线程的多步操作相互穿插：

```text
生产者 A：next() -> 0
生产者 B：next() -> 1
生产者 B：write(1)
生产者 B：publish(1) -> cursor=1
生产者 A：write(0)
生产者 A：publish(0) -> cursor=0
```

所有单次读写都可以是合法的，最终业务状态却是错误的：cursor 从 1 倒退到了 0。

消费者的代码又把 cursor 当作扫描上界：

```java
long available = sequencer.cursor();
while (next <= available) {
    if (ring.available.isAvailable(next)) {
        consume(next);
    }
}
```

当消费者准备处理 `next=1` 时，如果 cursor 已经被写回 0，那么 `1 <= 0` 为 false。即使 `isAvailable(1)` 明明是 true，消费者也根本没有机会检查它。

这揭示了一个非常重要的区别：

```text
一次写入是原子的
        ≠
写入的值符合业务约束
        ≠
一组状态转换构成正确的并发协议
```

## 3. 并发设计的第一原则：先写不变量

不变量是指：不管线程按照什么顺序交错执行，只要系统仍处于合法状态，这个条件就必须成立。

对于当前 RingBuffer，至少存在以下不变量：

| 不变量 | 含义 | 被破坏后的结果 |
|---|---|---|
| 申请序号唯一且单调递增 | 两个生产者不能得到同一个 sequence | 同一逻辑位置被重复写入 |
| 不能覆盖未消费槽位 | `sequence - bufferSize` 必须不超过消费者进度 | 消费者读到后续轮次的数据 |
| 发布上界不能倒退 | cursor 必须是单调上界 | 已发布序号从消费者扫描范围消失 |
| 数据写入先于发布标记 | 消费者看到 available 时，Entry 必须已经完整 | 读取 null、旧值或半完成状态 |
| 消费者按序推进 | sequence=N 未完成时不能越过 N | 顺序语义被破坏，槽位过早释放 |
| 槽位只能按代复用 | 相同 slot 的不同环次不能被混淆 | 位图出现 ABA 式误判 |
| 系统始终保留推进路径 | 等待关系不能形成闭环 | 无锁代码仍然逻辑死锁 |

先写不变量有三个好处。

第一，它把模糊的“应该线程安全”变成可以检查的条件。例如：

```text
错误说法：cursor 应该是线程安全的
准确说法：多个生产者并发更新后，cursor 必须始终等于或大于它曾经出现过的最大值
```

第二，它能帮助我们选择正确工具。若要求 cursor 只增不减，普通 volatile set 明显不够；需要改变 cursor 的职责，或者使用 CAS 实现单调更新。

第三，它能直接指导测试。测试不应该只断言“程序跑完”，还应持续检查 cursor 是否倒退、消费值是否等于期望 sequence、生产者是否覆盖未消费槽位。

### 3.1 不变量不是注释，而是设计约束

很多并发 bug 的根源不是程序员不知道某个 API，而是代码中从未明确写出真正的约束。

例如：

```java
cursor.set(sequence);
```

如果 cursor 的定义只是“最近一次发布的序号”，这行代码没有问题；如果消费者把 cursor 理解成“所有值得检查的序号上界”，这行代码就必须保证单调。

所以同一个字段的语义必须是唯一且稳定的。不能让生产者把它理解成“最后完成的序号”，消费者却把它理解成“最大可扫描上界”。

## 4. 把并发操作看成状态机，而不是几行代码

RingBuffer 中的一个逻辑 sequence，不是从“不存在”瞬间变成“可消费”。它会经历一系列状态：

```text
FREE
  ↓ next() 成功
CLAIMED(sequence)
  ↓ 写入 Entry
WRITTEN(sequence)
  ↓ publish / available 标记
PUBLISHED(sequence)
  ↓ 消费者处理
CONSUMED(sequence)
  ↓ consumerCursor 推进，允许下一轮复用
FREE(next generation)
```

从状态机角度看，每条边都有前置条件和后置条件：

- `FREE -> CLAIMED`：必须还有容量，且申请序号不能重复
- `CLAIMED -> WRITTEN`：只有获得该 sequence 的生产者能写对应槽位
- `WRITTEN -> PUBLISHED`：数据写入必须 happens-before 发布状态被消费者观察
- `PUBLISHED -> CONSUMED`：消费者只能按序进入，不能越过空洞
- `CONSUMED -> FREE`：只有消费进度推进后，生产者才能复用该槽

这种表达方式比直接阅读循环更容易发现问题，因为我们会自然追问：

- 有没有跳过某个状态？
- 有没有两个线程同时认为自己拥有同一状态？
- 发布信号是否可能早于数据完成？
- 下一代状态是否可能被误认为上一代？
- 某个状态是否存在进入后再也无法离开的路径？

### 4.1 线性化点：一个操作到底何时算完成

在并发算法中，一个由多行代码组成的操作，通常需要找到一个可以被视为“瞬间生效”的位置，这就是线性化点。

例如：

- 申请 sequence 的线性化点可以是 `nextValue.compareAndSet(current, next)` 成功
- 发布 sequence 的线性化点可以是 available 状态从未发布变成已发布
- 释放槽位的线性化点可以是 consumerCursor 推进到该 sequence

如果我们无法回答一个操作的线性化点在哪里，就很难判断其他线程什么时候可以依赖该操作的结果。

本次案例中，cursor 的职责混乱，本质上也是线性化语义混乱：`publish(1)` 已经完成后，后续 `publish(0)` 居然能让系统看起来像只发布到了 0。这说明 cursor 并不是可靠的发布完成线性化点。

## 5. 变量安全、操作安全和协议安全是三个层次

### 5.1 变量安全

变量安全关心单个共享变量的读写：

- 是否会发生数据撕裂？
- 更新是否原子？
- 一个线程能否看到另一个线程的写入？

`AtomicLong`、volatile 和 VarHandle 经常用于解决这一层问题。

### 5.2 操作安全

操作安全关心由多次读写组成的完整动作：

```java
if (stock > 0) {
    stock--;
}
```

即使 `stock` 的单次读写可见，整个“检查并扣减”也不是原子的。两个线程可能都看到库存为 1，然后都成功扣减。

### 5.3 协议安全

协议安全关心多个角色之间如何协作：

```text
生产者：申请 -> 写入 -> 发布
消费者：等待 -> 校验发布 -> 消费 -> 释放
```

即使每个角色内部的操作都正确，双方对同一个字段的语义理解不一致，协议仍会失败。

当前案例可以概括为：

```text
nextValue 的变量更新是安全的
availableBuffer 的位更新是安全的
cursor 的单次 volatile 写也是安全的

但“申请—写入—发布—扫描—消费—释放”的整体协议不安全
```

所以需要记住：

> 变量安全不等于操作安全，操作安全不等于协议安全，协议安全也不自动等于系统能够持续推进。

## 6. 安全性之外，还要证明系统能够继续推进

并发正确性至少包含两大方面。

### 6.1 安全性：坏事不会发生

安全性关注：

- 不会重复分配 sequence
- 不会覆盖未消费槽位
- 不会消费未发布数据
- cursor 不会倒退
- 不会把下一轮槽位误认为当前轮

这些问题通常表现为错读、重复、丢失、越界或状态不一致。

### 6.2 活性：好事最终会发生

活性关注：

- 已发布的数据最终会被消费
- 等待容量的生产者最终能够继续
- 系统不会永久停在某个中间状态
- 超时和取消机制真的能够被执行

当前 bug 是一个很好的活性案例：没有线程持有互斥锁，却仍然发生逻辑死锁。

```text
消费者等待 cursor 至少达到 next
        ↓
cursor 倒退，消费者不再推进 consumerCursor
        ↓
生产者等待 consumerCursor 释放环形容量
        ↓
所有生产者无法完成新的 publish 来抬高 cursor
```

双方都在等待对方先推进，但没有任何一方还保留推进路径。

### 6.3 无锁不等于不会死锁

“无锁”通常只表示算法不依赖传统互斥锁，或者满足某种非阻塞进度保证。它不表示业务协议天然不会卡住。

下面这些代码都可能没有 `synchronized` 或 `Lock`，却仍然永久等待：

- 自旋条件永远无法变成 true
- 两个进度变量互相等待
- 发布信号丢失或倒退
- 所有工作线程都在等待一个只能由它们自己触发的事件
- 主线程先无限 join，再把超时检查写在 join 后面

因此，审查并发算法时必须单独画出等待关系：

```text
谁在等待什么？
这个条件由谁改变？
负责改变它的线程现在还能运行吗？
如果该线程恰好暂停，是否还有其他线程能推进系统？
```

## 7. 从本次案例总结出的五类并发陷阱

### 7.1 控制流看起来在等待，实际仍执行了更新

旧版 `next()` 使用：

```java
do {
    if (capacityNotEnough) {
        continue;
    }
} while (!nextValue.compareAndSet(current, next));
```

这里的 `continue` 会跳到 `while` 条件，而 CAS 正好写在条件里。容量不足时，本应放弃本轮申请，程序却仍可能执行 CAS 并拿到不该得到的序号。

教训不是简单的“不要使用 do-while”，而是：

> 并发算法的控制流本身也是协议的一部分。每一个 continue、break、异常和提前返回，都必须确认不会绕过关键前置条件。

### 7.2 修复了观察症状，却没有恢复底层不变量

消费者最初卡在 `Entry.sequence != next`。加入 availableBuffer 后，消费者不再依赖会被覆盖的 Entry.sequence，程序看起来不再卡住。

但如果生产者仍能覆盖未消费槽位，available 位只能证明“某个 sequence 曾经发布过”，不能证明槽里现在保存的还是它的数据。

这类修复的危险在于：

```text
错误症状消失
    ≠
根因被修复
    ≠
不变量重新成立
```

修复后必须重新检查最初的不变量，而不是只重复原来的表面测试。

### 7.3 可见性正确，不代表复合更新不会丢失

普通 `long[]` 用读—改—写设置 bit：

```java
bits[index] = bits[index] | mask;
```

两个生产者同时设置同一个 long 中的不同 bit，可能互相覆盖结果。即使消费者最终能够看到数组写入，也可能看到的是已经丢了一位的结果。

AtomicLongArray 的价值不只是“让其他线程看见”，还在于通过 CAS 保证复合更新不会丢失。

### 7.4 原子覆盖正确，不代表值的变化方向正确

`cursor.set(sequence)` 是合法的 volatile 写，但业务要求 cursor 单调不回退。

```text
原子性保证：其他线程不会读到半个 long
业务要求：其他线程不能在读过 100 后又读到 99
```

底层原子性没有包含“取最大值”的业务语义。这必须由算法显式保证。

### 7.5 诊断机制放错位置，会让真正的卡死不可见

当前 main 先执行：

```java
for (Thread producer : producers) {
    producer.join();
}
```

然后才进入 10 秒超时循环。如果生产者已经卡在容量等待中，主线程永远到不了超时判断。

监控代码也属于并发协议的一部分。必须验证：

- 超时是否覆盖所有可能阻塞的位置？
- 发生卡死时，负责打印诊断信息的线程是否还能执行？
- 计数器本身是否线程安全，还是会丢更新并给出假数据？
- 日志中的进度代表“申请”“发布”还是“消费完成”？

## 8. 一套可复用的并发分析方法

下面的方法适用于 RingBuffer，也适用于库存扣减、订单状态机、缓存刷新、任务调度和分布式消息处理。

### 第一步：列出共享状态及其唯一语义

建议先写一张表：

| 状态 | 谁写 | 谁读 | 精确定义 | 是否允许回退 |
|---|---|---|---|---|
| nextValue | 多个生产者 CAS | 生产者 | 最大已申请序号 | 否 |
| producer cursor | 生产者 | 消费者 | 消费者可扫描的上界 | 否 |
| available | 对应生产者 | 消费者 | 某个具体 sequence 是否发布 | 按槽位代次变化 |
| consumerCursor | 单个消费者 | 生产者 | 最大连续已消费序号 | 否 |
| Entry.value | 获得槽位的生产者 | 消费者 | 当前 sequence 对应数据 | 只能在获得复用权后覆盖 |

如果一个字段无法用一句话定义，或者不同线程对它有不同理解，设计很可能已经存在问题。

### 第二步：写出必须始终成立的不变量

不要只写“线程安全”，而要写成可判断的表达式：

```text
nextValue(t2) >= nextValue(t1), 当 t2 晚于 t1
consumerCursor <= claimedCursor
claimedSequence - bufferSize <= consumerCursor，复用槽位之前
available(sequence) == true 时，对应 value 已完整写入
consumerCursor=N 表示 [0..N] 全部已经消费
```

### 第三步：画出状态转换并标记线性化点

对每个操作写清：

- 前置状态
- 修改了哪些字段
- 操作完成的线性化点
- 其他线程从何时可以依赖它
- 失败或暂停后谁负责恢复

### 第四步：主动构造最坏线程交错

不要只想最自然的执行顺序。刻意让线程停在最危险的位置：

```text
A 申请较小序号后暂停
B 申请较大序号并先发布
A 恢复并后发布
消费者恰好在 cursor 倒退后重新读取上界
```

常用问题包括：

- 如果线程在这行之后暂停十秒会怎样？
- 如果另一个线程把后续步骤先做完会怎样？
- 如果最后一次写入来自较旧操作会怎样？
- 如果没有下一次事件来纠正当前状态会怎样？

### 第五步：分别证明安全性和活性

安全性证明：所有交错都不会产生非法状态。

活性证明：在合理调度假设下，系统不会永久停在中间状态，并且等待条件最终有线程能够改变。

### 第六步：最后才映射到并发工具

明确需要的保证后再选择工具：

- 只需要跨线程可见：volatile
- 需要单变量原子读改写：CAS、Atomic、VarHandle
- 多字段必须作为整体提交：锁、不可变快照、版本化状态或事务
- 需要线程等待条件：Condition、Semaphore、队列或明确的等待策略
- 需要跨进程一致性：数据库事务、唯一约束、幂等键、消息协议

### 第七步：让监控直接观测不变量

与其只打印吞吐量，不如记录：

- cursor 是否曾经倒退
- 当前 next 的 available 位是否已经为 true
- claimed、published、consumed 三种进度分别是多少
- 每个线程在等待哪个条件
- 多久没有任何全局进度变化

好的诊断信息应直接回答“哪个不变量被破坏”，而不只是告诉我们“程序好像很慢”。

## 9. 面对强依赖的六种解决方向

当一个业务操作由多个步骤组成，而且步骤之间存在强依赖时，最直观的解决方案通常有两个：

1. 保留依赖，通过加锁或串行化强制执行顺序
2. 重新设计业务，让操作不再依赖执行顺序

这两个方向都正确，但还不完整。并发设计还可以从共享范围、状态所有权、冲突处理和失败恢复等角度入手。更完整的解决空间包括下面六种方向。

需要先明确一个边界：

> 如果某个顺序依赖来自真实业务语义，就不能为了并发性能直接删除它；只有人为设计出来、能够等价改写的依赖，才可以安全消除。

例如“数据必须写完才能发布”是真实依赖，不能解除；“多个发布者中最后一次写 cursor 的值代表全局进度”则是一种不可靠的实现设计，可以重新定义。

### 9.1 方向一：保留依赖，把状态转换强制变成有序整体

如果业务步骤必须严格遵循：

```text
检查前置条件 → 修改核心状态 → 写入关联状态 → 对外发布结果
```

那么这些步骤必须被保护为一个完整状态转换，使其他线程不能观察或制造非法的中间组合。

加锁是其中一种方式，但不是唯一方式：

- `synchronized` 或 `Lock` 保护临界区
- 数据库事务原子提交多条状态
- CAS 状态机只允许从预期旧状态转换到新状态
- 单线程执行器或事件循环按顺序处理命令
- Actor 让一个状态只由对应 actor 修改
- 消息队列按业务键分区，在分区内部顺序消费

这些方案的共同点不是“都使用锁”，而是：

> 为一组强依赖步骤建立明确的提交边界和唯一顺序。

例如订单的合法状态只能是：

```text
待支付 → 已支付 → 已发货 → 已完成
```

这种业务状态机不能简单改成任意顺序，只能明确允许的转换，并拒绝旧请求或非法转换。

### 9.2 方向二：消除顺序依赖，让操作可以交换、重复或合并

如果顺序依赖并非业务本质，可以尝试重新定义操作，使执行顺序不再影响最终结果。

常见目标包括：

- **交换性**：先执行 A 再执行 B，与先 B 后 A 结果相同
- **幂等性**：同一个操作重复执行，最终结果不变
- **结合性**：操作如何分组不影响最终结果
- **单调性**：状态只能朝一个方向发展，不会被旧结果回退

当前 cursor 问题就是一个直接例子。错误操作是：

```text
cursor = sequence
```

它依赖生产者的完成顺序：先发布 1、后发布 0，cursor 就会倒退成 0。

如果 cursor 的业务语义允许定义成最大上界，那么更新可以改写为：

```text
cursor = max(cursor, sequence)
```

此时：

```text
max(max(-1, 1), 0) = 1
max(max(-1, 0), 1) = 1
```

两个发布顺序得到相同结果。算法不再依赖“谁最后写入”，并发复杂度就会明显降低。

不过 `max` 只解决 cursor 单调问题，不能代替具体 sequence 的发布标记。消费者仍然需要 available 状态判断中间是否存在空洞。这再次说明：重新设计一个操作时，必须同时检查完整协议的不变量。

### 9.3 方向三：消除或减少共享状态

这是非常有意义、也往往最有效的方向：

> 不共享，就不需要同步；少共享，就少协调。

可以采用以下方法。

#### 使用线程本地状态

每个线程先独立计算或累计，最后再合并。例如统计计数时，让每个线程维护局部计数，而不是每次更新都竞争同一个全局变量。

#### 使用不可变数据

线程不原地修改共享对象，而是创建新版本，再一次性替换完整快照。读线程看到的始终是某一个完整版本，不会看到跨多个字段的中间状态。

#### 按业务键分区

按用户、订单、账户或分片键拆分状态：

```text
订单 A → 分区 0
订单 B → 分区 1
订单 C → 分区 0
```

同一个订单在分区内保持顺序，不同订单可以并行。这样可以把全局共享竞争缩小成局部竞争。

#### 让结果可以独立合并

如果各线程能够生成独立结果，最终通过 sum、max、集合并集等操作合并，就不必在每个中间步骤上共享可变状态。

消除共享通常比优化锁更有价值，因为它不是让竞争更便宜，而是直接减少竞争发生的机会。

### 9.4 方向四：限制共享状态的所有权

有时状态无法消除，但可以规定它只有一个写者：

```text
多个业务线程
      ↓ 发送命令
单一状态所有者
      ↓ 修改状态
消费者或下游
```

这就是单写者、Actor、事件循环和消息传递模型的共同思想。

其他线程可以读取快照或提交命令，但不能直接修改状态。这样能够明确：

- 谁拥有状态
- 谁能够执行状态转换
- 转换按照什么顺序发生
- 操作在哪个时刻完成
- 失败后由谁负责恢复

Disruptor 的单生产者模式之所以简单而高效，也与单写者原则有关：同一个生产进度只有一个线程修改，就不需要为每次申请序号进行多生产者 CAS 竞争。

“限制共享所有权”和“消除共享”很接近，但并不相同：前者仍有共享数据，只是把修改权集中到一个明确角色；后者尽量让数据本身不被多个线程共同访问或修改。

### 9.5 方向五：乐观并发，允许同时工作但在提交时检测冲突

如果完全串行的代价太高，而且真实冲突概率较低，可以让多个线程先并发计算，在提交时验证前提是否仍成立：

```text
读取 state，version=10
执行本地计算
提交时检查 version 是否仍为 10
    是 → 提交新状态，version=11
    否 → 放弃、重试或报告冲突
```

常见实现包括：

- CAS 循环
- 数据库乐观锁
- 版本号和条件更新
- MVCC
- compare-and-set 状态机

它的核心不是“没有锁”，而是把并发冲突变成一种显式结果：

> 没有冲突时并发完成；发生冲突时必须检测出来，并进入明确的重试、失败或合并路径。

乐观并发不能省略冲突处理。如果 CAS 失败后没有正确重读状态，或者数据库更新影响行数为 0 却仍然向下执行，业务协议依然会被破坏。

### 9.6 方向六：允许中间失败或乱序，但设计检测与恢复

跨数据库、消息系统和远程服务的操作，经常无法放进一个本地原子临界区。这时不能假设每个步骤永远成功，而要让协议能够从部分完成中恢复。

常见机制包括：

- 幂等键和重复请求去重
- 有界重试与退避
- 超时、租约和任务接管
- 补偿事务和 Saga
- Outbox 保证业务状态与待发送事件一起落库
- 定期对账和状态修复

这个方向接受：系统可能暂时处于中间状态、消息可能重复、网络可能超时。但它要求：

- 中间状态能够被识别
- 重复执行不会制造新的错误
- 必须有角色负责重试或补偿
- 最终能够回到合法状态

它可以称为容错型并发协议：不要求错误永远不发生，而要求错误发生后仍然保留恢复路径。

### 9.7 四个层次和推荐选择顺序

上面的六种方向可以进一步归纳成四个层次：

| 层次 | 核心问题 | 典型方案 |
|---|---|---|
| 避免共享 | 能不能根本不让多个线程修改同一状态 | 线程本地、不可变对象、独立结果合并 |
| 限制共享 | 能不能缩小共享范围或只保留一个写者 | 分区、单写者、Actor、消息传递 |
| 协调共享 | 必须竞争时，怎样保证合法状态转换 | 锁、CAS、事务、版本号 |
| 容忍冲突 | 无法完全原子时，怎样发现和恢复 | 幂等、重试、补偿、Outbox、对账 |

设计时可以按照下面的顺序追问：

```text
这个状态真的需要共享吗？
        ↓ 是
能不能让它只有一个所有者或按业务键分区？
        ↓ 不能
操作能不能改造成交换、幂等、单调或可合并？
        ↓ 不能完全做到
哪些步骤必须原子提交，应该用锁、CAS 还是事务？
        ↓ 无法跨系统原子提交
怎样检测部分失败，并通过重试、补偿或对账恢复？
```

这不是绝对的性能优先级，而是一种降低并发复杂度的设计优先级。越靠前的方案，通常越能从结构上减少需要证明的线程交错。

### 9.8 思想纲领为什么比单纯背加锁技巧更重要

思想纲领决定“系统需要什么保证”，具体并发细则决定“代码是否真的提供了这个保证”。

例如：

```text
思想纲领：cursor 必须单调不回退
实现细则：普通 volatile set 不能实现 max 语义
```

```text
思想纲领：数据写完以后才能发布
实现细则：写 Entry 和设置 available 之间必须建立正确的 happens-before
```

```text
思想纲领：多个生产者设置不同 bit 不能丢失
实现细则：普通 long[] 的读—改—写不是原子的，需要 CAS 或等价机制
```

因此，不能把两者理解成“掌握思想以后就不需要学习细则”。更准确的关系是：

> 思想决定要维护什么，细则决定实现是否真的维护住了它。

只背技巧，容易写出“到处都有 Atomic，但协议仍然错误”的代码；只懂纲领却不了解 Java 内存模型，也可能设计正确、实现错误。

真正成熟的并发设计，会先用不变量和状态机确定目标，再用锁、CAS、volatile、事务或消息协议精确落实这些目标。

## 10. 如何设计真正能抓住问题的并发测试

### 10.1 测试不变量，不只测试最终结果

“程序在十秒内结束”是一个很弱的断言。它无法发现偶发错读，也无法区分死锁和机器变慢。

更有价值的断言包括：

```text
每次消费的 event 都等于当前 sequence
cursor 从未小于它的历史最大值
每个 sequence 恰好消费一次
任何槽位被复用前，对应旧 sequence 已消费
最终 claimed == published == consumed == total - 1
```

### 10.2 用最小数据构造确定性乱序

并发测试不一定要依赖百万次随机碰撞。cursor 倒退只需要两个序号：

```text
申请 0
申请 1
发布 1
发布 0
断言 cursor 仍然 >= 1
```

再启动消费者，断言它能消费 0 和 1。这个测试比跑 1.5 亿条消息更快，也更稳定地命中根因。

可以使用 CountDownLatch、CyclicBarrier 或测试专用钩子，精确控制线程在申请、写入和发布之间暂停。

### 10.3 用小 RingBuffer 强制绕环

将 bufferSize 设置为 2、4 或 8，可以快速触发：

- 容量保护
- 槽位复用
- 慢消费者
- available 状态代次问题
- 生产者等待消费者释放容量

小模型更容易穷举线程交错，也更容易解释失败现场。

### 10.4 超时必须包围整个并发生命周期

不要先无限 join，再在后面检查超时。测试应该给生产、消费和线程退出整个流程设置期限。

超时时需要输出：

- 所有线程状态和栈
- claimed、cursor、consumerCursor
- consumerNext 及其 available 状态
- 每个生产者最后申请或发布的 sequence

### 10.5 压力测试不能替代确定性测试

压力测试适合发现未知竞态和性能退化，但有三个局限：

- 失败难以稳定复现
- 跑完不代表覆盖了危险交错
- 大数据量可能把算法错误和单纯性能问题混在一起

最佳组合是：

```text
确定性时序测试：证明已知不变量
小容量循环测试：高频触发边界条件
随机压力测试：探索未知交错
长时间稳定性测试：观察资源和进度
```

## 11. 锁、CAS、volatile 和 Atomic 到底解决什么

| 工具 | 主要保证 | 不会自动保证 |
|---|---|---|
| volatile | 可见性、特定有序性、单次读写语义 | 复合操作原子性、业务值单调、多个字段一致 |
| CAS | 单变量条件更新的原子性 | 多变量事务、公平性、一定成功、协议语义正确 |
| AtomicLongArray | 数组元素的原子更新与可见性 | 不同元素之间的整体一致性、槽位代次语义 |
| synchronized / Lock | 临界区互斥及 happens-before | 锁顺序正确、不会死锁、业务不变量自动成立 |
| park / wait / Condition | 线程等待与唤醒机制 | 等待条件最终成立、不会丢失推进路径 |
| 线程安全集合 | 集合自身操作安全 | 多个集合或集合外字段组成的业务事务安全 |

工具没有错，错误通常来自我们把工具提供的保证扩大解释了。

例如：

```text
volatile 让 cursor 可见
```

不能推导出：

```text
cursor 一定单调
cursor 一定代表连续发布进度
消费者一定能够完成
```

正确的问题不是“这里该不该用 AtomicLong”，而是：

> 这个状态需要满足什么语义？哪个操作负责建立该语义？这个工具提供的保证是否刚好覆盖它？

## 12. 这套思想如何迁移到普通业务系统

### 12.1 库存扣减

业务不变量：库存不能小于 0，同一订单不能重复扣减。

仅把库存字段改成 AtomicInteger，不一定解决“校验订单—检查库存—扣减—记录流水”的整体原子性。需要定义完整事务边界和幂等语义。

### 12.2 订单状态流转

业务不变量：订单不能从已退款回到已支付，状态版本不能倒退。

这与 cursor 倒退非常相似。单次数据库 UPDATE 是原子的，不代表旧请求晚到后不会覆盖新状态。通常需要条件更新：

```sql
UPDATE orders
SET status = ?, version = version + 1
WHERE id = ? AND version = ? AND status = ?;
```

### 12.3 数据库与消息双写

业务不变量：业务状态提交后，对应事件最终必须发送；事件不能引用未提交数据。

“先写数据库再发消息”由两个独立步骤组成，任何一步之间崩溃都会破坏协议。Outbox 模式的价值，就是把两个系统之间的多步骤操作重新组织成可恢复的状态机。

### 12.4 缓存与数据库一致性

业务不变量：旧请求不能在新请求之后把旧值重新写回缓存。

即使缓存 set 和数据库 update 各自是线程安全的，两个请求乱序完成仍可能发生类似 cursor 倒退的“旧值覆盖新值”。需要版本号、失效策略或明确的写入协议。

### 12.5 任务调度与重试

业务不变量：任务最终完成，但同一次副作用不能被无限重复执行。

重试提高活性，却可能破坏安全性；去重保证安全性，却可能让错误状态永久无法推进。必须同时设计幂等键、状态转换、租约超时和恢复责任。

这些场景使用的技术不同，但分析问题的骨架完全相同：

```text
共享状态是什么？
不变量是什么？
完整操作分成哪些步骤？
步骤之间在哪里可能失败或乱序？
谁负责恢复？
系统是否始终存在推进路径？
```

## 13. 最终总结：并发编程的设计图

这次多生产者问题最有价值的地方，不是记住某一行 `continue` 或把某个字段换成 Atomic，而是形成了下面这套思考方式。

### 13.1 六句核心结论

1. **单个变量线程安全，不代表复合操作线程安全。**
2. **复合操作线程安全，不代表多个角色组成的并发协议正确。**
3. **先定义业务不变量，再选择锁、CAS、volatile 和 Atomic。**
4. **并发正确性既要保证不出错，也要保证系统最终能够推进。**
5. **修复后要验证不变量是否恢复，不能只看原来的症状是否消失。**
6. **测试应主动制造最坏交错，而不是期待压力运行偶然撞中它。**

### 13.2 审查并发代码时的最终清单

看到一段并发代码，可以依次追问：

- 共享状态有哪些？每个状态由谁写、谁读？
- 每个字段是否只有一个准确的业务含义？
- 哪些不变量必须在所有线程交错下成立？
- 一个业务操作由哪些步骤组成？线性化点在哪里？
- 线程可以在哪一步暂停？另一个线程会怎样穿插？
- 是否可能出现旧操作覆盖新状态？
- 发布信号与真实数据之间是否存在 happens-before？
- 等待条件由谁改变？负责改变它的线程是否仍能运行？
- 是否同时证明了安全性和活性？
- 测试是在验证不变量，还是只验证“这一次碰巧跑完”？

### 13.3 真正应该记住的一句话

> 并发编程真正困难的不是让多个线程同时运行，而是让它们在任意合法交错下，仍然共同维护同一套业务事实，并且始终有人能够把系统推进到下一个合法状态。

这就是本次案例带来的最重要收获：从“某个变量该用什么并发关键字”，上升到“整个系统正在执行什么状态协议”。当我们开始用不变量、状态机、线性化点和可推进性理解并发时，才算真正找到了分析多线程问题的主线。
